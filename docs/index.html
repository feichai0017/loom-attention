Rust · control plane · LLM serving

QuillCache

A vendor-neutral KV-cache control plane and evaluation platform for LLM inference. It does not run models and it does not move KV tensors — it sits in front of real engines (vLLM / SGLang) and owns the metadata and the decisions: block identity, residency, cache-aware routing, and identity-governed safe reuse.

Two modes — an OpenAI-compatible online gateway, and an experiment harness that replays one trace across pluggable engines × routing policies × index backends to measure them apples-to-apples.

Where it sits

QuillCache is the control/decision plane. The engines run the model; the data plane (LMCache / KVBM / FlexKV) moves the KV tensor bytes. The two planes meet at the index and the storage tier, never as the same object.

Honest scope: QuillCache is a research/portfolio prototype, not a drop-in for Dynamo. It re-implements the control-plane ideas (and adds an identity-safe-reuse primitive and a persistent ART index study) at a size one person fully understands and can measure.

The storage angle — ART vs LSM for the residency index

The residency index is written on every KV event and read on every request (longest reusable prefix). Most systems keep it in memory; QuillCache makes it a pluggable, persistent, measured component and asks which engine fits. Same workload, same IndexBackend trait, via quillcache bench-index:

backendprefix-scan p50eviction churnrecoveryon-disk
memory (flat map)O(N)1.15M/s—0
rocksdb (LSM)2.7 µs154k/s2.0 ms175 KB
holt (ART)1.2 µs403k/s0.67 ms8.4 MB

ART wins prefix-scan latency and recovery; LSM wins disk footprint. The eviction numbers are after a secondary block-hash index fixed an O(scope) remove_block the churn benchmark exposed (100–300× on the persistent backends). holt is the user's own persistent ART/WAL engine, here serving as the prefix-native index.

Identity-governed safe reuse

A KV block's content hash is computed from its tokens — so the same tokens collide across identities (tenant, LoRA adapter, model/quant, tokenizer version) while the KV tensors do not. A cache that reuses on content alone serves wrong or leaked state. QuillCache makes the reuse contract explicit and enforces it inline on the live gateway.

# tenant-a caches a prefix; tenant-b asks for the same content
$ curl ... -H 'tenant: b' /v1/chat/completions -i
x-quillcache-local-hits: 0          # refused — not a stale-but-safe hit
x-quillcache-reuse-refused: 2       # 2 cross-tenant blocks the guard blocked
# gateway log: WARN identity guard refused unsafe cross-identity reuse
  • Privacy: cross-tenant reuse is a prefix-cache leak — the guard blocks it.
  • Correctness: cross-adapter / model / tokenizer reuse is numerically wrong — blocked.
  • Precise, not blunt: same-identity reuse is preserved; overhead is ~1.7% on a realistic workload.

Try it

# Experiment mode — replay one trace across policies × index backends
$ cargo run --features holt -- bench-index --backend holt --churn-ops 500
$ cargo run -- safe-reuse            # naive content reuse vs the identity guard

# Online mode — OpenAI-compatible gateway, persistent ART residency index
$ cargo run --features holt -- gateway --config gateway.yaml   # index: holt
$ curl localhost:8080/v1/chat/completions -d '{...}'  # x-quillcache-* decision headers
